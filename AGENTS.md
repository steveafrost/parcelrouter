## App Overview

**ParcelRouter** is a local-first, self-hosted package tracking application that routes delivery events from your email inbox. It polls iCloud Mail via IMAP, extracts tracking numbers using regex pattern matching, scores detections by confidence (high/medium/low), and stores results locally in SQLite. High-confidence detections are auto-created as tracked packages; medium/low-confidence ones go to a review queue for manual approval. Optionally syncs to the Parcel app API and/or emits webhook events.

**GitHub:** https://github.com/steveafrost/parcelrouter  
**Domain:** https://parcelrouter.com  
**License:** MIT  
**Language:** TypeScript (Node.js)  
**Deployment:** Docker Compose (single container)

### Data Flow

1. **Scheduler** fires on a cron interval (default: every 30-60 min)
2. **EmailPoller** connects to iCloud via IMAP, fetches emails since last poll
3. **EmailParser** filters by shipping keywords, extracts tracking via regex
4. **TrackingPatterns** scores confidence: UPS/Amazon/USPS formats = high, FedEx/DHL with context = medium, ambiguous = low
5. **High confidence** → directly into `tracked_packages` table
6. **Medium/Low confidence** → `review_items` table with status='pending'
7. **WebhookDispatcher** fires events for package.created, review.created, etc.
8. **ParcelClient** optionally syncs to Parcel API (POST /add-delivery)
9. **Dashboard** polls `/health`, `/stats`, `/packages`, `/review` every 30s

## Current State

### What's Working Well

- **Solid unit test coverage** — 12 test files with ~47 individual tests across all layers
- **E2E coverage** — Playwright tests cover dashboard flows and marketing site
- **Confidence-based routing** — intelligent triage: high confidence auto-creates, med/low waits for review
- **Carrier support** — UPS, FedEx, USPS, Amazon Logistics, DHL, OnTrac, LaserShip
- **Retailer extraction** — 15+ known retailer domains + domain-name-to-brand fallback
- **Order number rejection** — explicit exclusion patterns prevent order numbers from being misdetected as tracking numbers
- **Multi-folder polling** — INBOX + Archive with per-folder last-poll tracking
- **Docker deployment** — multi-stage build, HEALTHCHECK, volume persistence
- **Guided installer** — `install.sh` prompts for credentials, validates, starts
- **Preflight checks** — validates config before the app runs
- **Webhook system** — typed event names, HMAC signing, timeout/abort
- **Stats tracking** — emails read, packages created counters
- **Graceful per-email error handling** — one bad email doesn't crash the poll

### What Needs Work

#### Critical Gaps

1. **No CI/CD pipeline** — No GitHub Actions, no automated test runner on push/PR. Tests only run manually. This is the single biggest operational gap.

2. **No API authentication** — The dashboard/REST API has no auth at all. Any process on the same network can read/delete packages and approve/ignore reviews. For a local-first app this is acceptable in Docker's default network, but a simple API key or bearer token would improve security.

3. **No graceful shutdown** — `SIGTERM`/`SIGINT` handlers are missing. IMAP connections, scheduler, and server don't clean up on shutdown.

4. **Unused `tracking_events` table** — The table exists in schema.sql with a foreign key to `tracked_packages`, but nothing in the code writes to it. Parcel events are never stored locally.

5. **No retry for Parcel sync failures** — When Parcel API fails (rate limit, network), the error is logged but the package is never retried. With a 20 req/day limit, this is especially painful.

#### Moderate Gaps

6. **Dashboard is a 1152-line monolith** — All CSS, HTML, and JS in one file. No bundler, no type checking, no component structure. Hard to maintain or extend.

7. **No structured logging** — Uses `console.log`/`console.error` everywhere. No log levels, no structured JSON, no centralized logger. Hard to monitor or debug in production.

8. **No input validation on API** — Express endpoints trust the body/params directly (except `id` via route param). No schema validation for POST/DELETE operations.

9. **No package status tracking** — The `package.created` event is emitted but there's no way to update a package's status later (delivered, in transit, exception). Parcel event webhook ingestion isn't implemented.

10. **Parcel API has no quota management** — The 20 req/day limit is documented but not enforced client-side. No retry-after header handling.

11. **No CORS configuration** — Express has no CORS middleware. Fine for single-container, but blocks any cross-origin use.

12. **No `updatePackage` API** — No endpoint to update package metadata (retailer, product name, etc.).

#### Minor Gaps

13. **Only iCloud IMAP** — Hardcoded to `imap.mail.me.com`. Gmail/Outlook/Generic IMAP is architecturally possible but untested and undocumented.

14. **No `.env` validation in `index.ts`** — Only IMAP_USER and IMAP_PASS are validated. PORT, IMAP_HOST, POLL_INTERVAL are used with defaults but no sanity checks at startup.

15. **Schema discovery is fragile** — `findSchemaFile()` tries 4 paths in `connection.ts`. This works but adds cognitive load for debugging path issues.

16. **No database migrations** — Schema is applied via `CREATE TABLE IF NOT EXISTS` only. Adding columns requires manual ALTER TABLE or delete-and-reset.

17. **No health check on IMAP connection** — `/health` checks the server is running but doesn't verify IMAP connectivity.

18. **Skipped integration tests are silent** — IMAP and Parcel integration tests are skipped by default with no warning output.

## Key Architectural Decisions

1. **SQLite over PostgreSQL** — Local-first, zero infrastructure. Single-container deployment. WAL mode for concurrent reads.

2. **IMAP polling over email forwarding** — Works with existing email setup. No need to configure forwarding rules. Downside: requires apple app-specific password.

3. **Confidence-based routing** — Not all tracking patterns are equally reliable. UPS/Amazon/USPS have very specific formats. FedEx/DHL are just 10-15 digit numbers that could be order IDs.

4. **Single HTML dashboard** — No frontend build step, no NPM for the frontend. Pragmatic for an MVP, but needs extraction as the app grows.

5. **No carrier API integration** — Relies purely on email parsing. No live tracking status. `tracking_events` table exists for future carrier API integration.

6. **Webhook-first automation** — Instead of building integrations directly, emit typed events to a configurable webhook. Users connect their own Home Assistant, n8n, etc.
